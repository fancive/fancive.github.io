分布式系统的一致性协议 raft / zab

发表于 2019-08-13 | 分类于 服务治理 ， 一致性协议

引言

如果需要构建大型的分布式系统，那么服务发现和配置管理是首先要考虑的问题，最近的几个面试中也经常被问到相关的问题，因此在这里总结一下，适用于分布式系统的配置管理/服务发现，或者其他 k/v 存储。

本文以raft和zab协议为例做比较。

关联

raft 和 zookeeper 的 ZAB 都是 leader-based 一致性协议，通过选举一个proposer作为leader降低多个proposer引起的冲突问题，大致都可以分为两个阶段

1）leader在时，由leader向follow同步日志

2）leader不在时，用选举算法选出一个新leader

leader在时

日志复制过程

raft 和 zab的处理流程基本一致

  1. client 连接leader(连接的是follower，则会转给leader)，这一点 raft 和 zab一致
  2. leader将请求转换成 entry(raft) 或者 proposal(ZAB)，并写入自己的日志中，得到日志index并发给follower
  3. follower接收到 entry 或者 proposal ， 会写入文件并回复OK
  4. leader如果接收到过半 OK，那么会更新leader的状态机，并回复客户端，此时
    1. ZAB会立刻向follower发送提交请求
    2. raft会更新leader的 commit index
  5. 对于follower来说
    1. ZAB接收到leader的提交请求会更新状态机
    2. raft会在下次心跳获取到leader 的 commit index 并更新自己的状态机

异常case

一旦leader发给follower的数据出现超时等异常

  1. raft:会不断重试，并且接口是幂等的
  2. Zab:follower会断开与leader之间的连接，重新加入该集群

leader不在时

节点有3种状态，leader , follower，candidate/looking , zab多一种不参与选举的follower角色被称为Observer

两种协议都有任期(朝代)的概念 term/epoch

  1. 选主就是选出朝代离现在最近(最大)的一个
  2. 如果朝代相同，那么选取拥有更长日志的为leader，所以选举广播每个候选人都会发送自己的<朝代、index>
  3. 如果<朝代、index>都相同那么ZAB会选机器编号最大的(一轮内重新投票, 会有一个子状态表示是一次选举)，raft会用新朝代选举

重新选举进入时机

raft

follower有一个选举时间，在该时间内如果未收到leader的心跳信息，则follower转变成candidate，自增term发起新一轮的投票，leader遇到新的term则自动转变成follower的状态

ZAB

leader和follower都有各自的检测超时方式，leader是检测是否过半follower心跳回复了，follower检测leader是否发送心跳了。一旦leader检测失败，则leader进入LOOKING状态，其他follower过一段时间因收不到leader心跳也会进入LOOKING状态，从而出发新的leader选举

日志同步

raft

Leader发生切换的时候，新Leader的日志和Follower的日志可能会存在不一致的情形。这时Follower需要对自身的日志进行截断处理，再从截断的位置重新同步日志。Leader自身的日志是Append-Only的，它永远不会抹掉自身的任何日志。
标准的策略是Leader当选后立即向所有节点广播AppendEntries消息，携带最后一条日志的信息。Follower收到消息后和自己的日志进行比对，如果最后一条日志和自己的不匹配就回绝Leader。
Leader被打击后，就会开始回退一步，携带最后两条日志，重新向拒绝自己的Follower发送AppendEntries消息。如果Follower发现消息中两条日志的第一条还是和自己的不匹配，那就继续拒绝，然后Leader被打击后继续后退重试。如果匹配的话，那么就把消息中的日志项覆盖掉本地的日志，于是同步就成功了，一致性就实现了。

zab

已经被处理的消息不能丢

当 leader 收到合法数量 follower 的 ACKs 后，就向各个 follower 广播 COMMIT 命令，同时也会在本地执行 COMMIT 并向连接的客户端返回「成功」。但是如果在各个 follower 在收到 COMMIT 命令前 leader 就挂了，导致剩下的服务器并没有执行都这条消息

由于所有提案被COMMIT 之前必须有大多数量的 Follower ACK，即大多数服务器已经将该 proposal写入日志文件。因此，新选出的Leader如果满足是大多数节点中proposal最多的，它就必然存有所有被COMMIT消息的proposal。

接下来，新Leader与Follower 建立先进先出的队列， 先将自身有而Follower缺失的proposal发送给 它，再将这些 proposal的COMMIT命令发送给 Follower，这便保证了所有的Follower都保存了所有的 proposal、所有的Follower 都处理了所有的消息

被丢弃的消息不能再次出现

当Leader 接收到消息请求生成 proposal后就挂了，其他Follower 并没有收到此proposal，因此新选出的Leader中必然不含这条消息。 此时，假如之前挂了的Leader 重新启动并注册成了Follower，它要与新的Leader保持一致，就必须要删除自己上旧的proposal

Zab 通过巧妙的设计 zxid 来实现这一目的。一个 zxid 是64位，高 32 是纪元（epoch）编号，每经过一次 Leader选举产生一个新的Leader，其epoch 号 +1。低 32 位是消息计数器，每接收到一条消息这个值 +1，新Leader 选举后这个值重置为 0

避免大的堆内存导致的gc开销

翻译自 Avoiding high GC overhead with large heaps

当分配的内存相对较小时，Go Garbage Collector（GC）运行得非常好，但是如果堆大小较大，GC最终可能会占用大量的CPU。 在极端情况下，它可能无法跟上程序的正常运行。

是什么问题?

GC的工作是确定可以释放哪些内存，并通过扫描内存来查找是否有内存指向这块内存。 简单地说，如果没有指向这块内存的指针，则这块内存可以释放。 这非常有效，但扫描的内存越多，所需的时间就越多。

假设您已经编写了基于内存的数据库，或者您正在构建一个需要大量查找表的数据管道。 在这些情况下，您可能会分配数Gb 的内存。 在这种情况下，您可能会失去相当多的潜在性能用于GC。

这是个大问题吗？

这些问题常见吗？ 我们来看看吧！ 这是一个很小的演示程序。 我们分配了十亿（1e9）个8字节指针，因此大约有8 GB的内存。 然后我们强制GC并计算需要多长时间。 我们这样做了几次来获得稳定的耗时。 我们还调用runtime.KeepAlive( ) 来确保GC /编译器在此期间不会丢弃我们分配的内存。

KeepAlive 将其参数标记为当前可达。这可以确保在调用 KeepAlive 的程序点之前，该对象不会被释放，并且其终结器不会运行。

GC会占用半秒钟的时间。为什么这会令人惊讶呢？我已经分配了10亿个指针。实际上每个指针检查的时间不到1纳秒。 查看指针的速度已经非常快了。

所以然后呢？

这似乎是一个根本问题。如果我们的应用程序需要在内存中维护一个大的查找表，或者如果我们的应用本身就是一个大的内存查找表，那么我们就遇到了一个问题。如果GC定期扫描我们分配的所有内存，我们将因为GC失去大量可用的处理能力。我们该怎么办？

我们基本上有两个选择。我们要么将内存隐藏从GC中隐藏，要么使GC对它不感兴趣。

使GC对它不感兴趣

记忆怎么可能是无趣的？嗯，GC正在寻找指针。如果我们分配的对象的类型不包含指针怎么办？GC还会扫描它吗？

我们可以试试。在下面的示例中，我们分配的内存大小与以前完全相同，但现在我们的分配中没有指针类型。我们分配了一个10亿个8字节int 的 slice，这也是大约8GB的内存。

对于完全相同大小的内存分配，GC的速度却要快1000倍。 事实证明，Go内存管理器知道每个被分配内存的类型，并会标记不包含指针的内存，以便GC不必扫描它们。 如果我们可以合理安排我们的内存表使之没有指针，那么我们就可以获得非常好的GC性能表现。

把内存从GC中隐藏起来

我们可以做的另一件事是向GC中隐藏我们分配的内存。如果我们直接向操作系统请求内存，GC永远不会发现它，因此不会扫描它。这比我们前面的例子要复杂一点！

问题的核心

因此，当我们在堆上分配大量内存时，以及当我们尝试通过将数据移动到我们自己的堆外分配来解决这个问题时，事实证明指针是问题所在。 如果我们可以避免我们分配内存中的任何指针，它们将不会导致GC开销，因此我们不需要使用任何堆外技巧。 如果我们使用堆外内存，那么我们需要避免存储指向堆分配的指针，除非这些指针也被GC可见的内存引用。

如何避免指针？

如果你有一个很大的堆，指针是必须避免的。但是你需要能够发现它们以避免它们，而且它们并不总是显而易见的。string、slice和 time.Time 都包含指针。如果你在内存中储存了大量的这些信息，可能需要采取一些步骤。

当我遇到大堆的问题时，主要原因如下。

  1. 很多string
  2. 使用time.time对象
  3. 带slice 作为value 的map
  4. 带 string 作为key的map

关于处理这些问题的不同策略有很多可以说。 在这篇文章中，我将谈谈处理字符串的一个想法。

Strings

什么是字符串？ 它有两个部分。 有字符串 header，告诉你字符串有多长，以及底层数据在哪里。 然后是基础数据，它只是一个字节序列。

将字符串变量传递给函数时，写入堆栈的是字符串header，如果类型是 []string，则它是slice 中出现的字符header。

字符串 header 由reflect.StringHeader描述，如下所示。

字符串header 包含指针，因此我们要避免存储字符串！

  1. 如果您的字符串只需要几个固定值，那么请考虑使用整数常量
  2. 如果您将日期和时间存储为字符串，则可能会解析它们并将日期或时间存储为整数
  3. 如果你从根本上需要保持很多字符串，那么请继续阅读……

假设我们存储了一亿个字符串。 为简单起见，我们假设这是一个巨大的全局var mystrings []字符串。

底层的mystrings是一个reflect.SliceHeader，看起来类似于我们刚刚看到的reflect.StringHeader。

对于mystrings，Len和Cap将各自为100,000,000，而Data将指向一个足够大的内存，足以容纳100,000,000个StringHeaders。 该内存包含指针，因此将由GC扫描。

字符串本身包括两部分。 此 slice 中包含的StringHeaders和每个字符串单独分配且不包含指针的Data。 从GC的角度来看，有问题是字符串 header，而不是字符串数据本身。 字符串数据不包含指针，因此不会扫描。 庞大的字符串 header数组确实包含指针，因此必须在每个GC周期进行扫描。

我们能做些什么呢？ 如果所有字符串字节都在单个内存中，我们可以通过偏移量跟踪内存中每个字符串的开头和结尾。 通过跟踪偏移量，我们的byte slice 中不再有指针，GC不再受到困扰。

我们通过这样做放弃的是为单个字符串释放内存的能力，并且我们增加了一些将字符串体复制到大 byte slice 的开销。

下面是一个演示这个想法的小程序。我们将创建100000000个字符串，将字符串中的字节复制到一个大byte slice 中，并存储偏移量。然后我们将显示gc时间仍然很短，并演示通过显示前10个字符串来检索字符串。

这里的原则是，如果不需要释放字符串，可以转换为更大的数据块中的索引，并避免有大量指针。如果你感兴趣的话，我已经建立了一个稍微复杂一点的东西，遵循这个原则。

线程 wiki

一条线程指的是进程中一个单一顺序的控制流。

同一进程中的多条线程将共享该进程中的全部系统资源，如虚拟地址空间，文件描述符和信号处理等等。但同一进程中的多个线程有各自的调用栈（call stack），自己的寄存器环境（register context），自己的线程本地存储（thread-local storage）

为什么需要熔断器

发表于 2019-05-13 | 分类于 技术方案

背景

  1. 我们采用的是同步(callee)阻塞(caller)的多进程模型
  2. 受限与db/cache 连接数，因此瓶颈一般在于worker数量，而不是硬件资源
  3. 我们服务逻辑上都在同一层(可以互相调用)，因此一个请求到后端各服务有乘数效应(放大)
  4. 服务健康失败会把不健康的节点踢除
  5. 还有一个外部变量是时任CTO希望收敛技术栈，有了php->golang的大背景

导致的p0问题 事故报告_2018.07.04（少数服务容量瓶颈拖慢了所有服务、导致雪崩，18年服务端最大的锅之一）

反思

  1. 提高硬件资源的利用率→改变同步阻塞的通信模型 （通过换golang解决）
  2. 避免请求到后端的乘数效应→ 对服务进程逻辑分层，减少同级服务的互相调用 （通过重新规划服务分层解决）
  3. 解决进程模型带来的连接数的问题→需要连接池(另外一个topic中会讨论)
  4. 如何在现有的通信模型下避免服务雪崩(一个服务性能瓶颈拖慢其他服务，导致雪崩)→熔断
  5. 修改健康检查失败踢节点的逻辑->只能主动将自己摘除

最后的实现方案如下，是一个单机系统的熔断器，熔断级别是针对单个service 级别(可以改为一个 service:endpoint / service:endpoint:function )

redis_list

redis 的链表节点是最简单的双向链表节点，表头和表尾都是NULL，没有环式链表。

链表容器中保存了表头和表尾，以及节点数量len，以及复制函数指针 void *(*dup)(void *ptr); 以及释放函数指针 void (*free)(void *ptr) 和值比较函数 int (*match)(void *ptr, void *key)

为什么这么设计

  • 使用void *value保存链表节点的值是为了存储多种数据类型
  • 链表容器中存储了 head 和 tail 是为了 O(1) 复杂度访问头尾节点
  • 存储len是为了在O(1)复杂度内获取链表长度
  • 三个函数指针是为了实现多态链表

横向比较

nginx 中有两个链表，单链表 ngx_list_t 和双向链表 ngx_queue_t

ngx_list_t

ngx_list_t 中的每个链表节点都是一个数组，数组元素数量上限存储在链表容器 nalloc 中，数组每个元素大小存储在链表容器 size 中，链表节点中只保存已有的元素数量 nelts ， 以及存放元素的开始地址 elts， 以及下一个链表节点指针 next
